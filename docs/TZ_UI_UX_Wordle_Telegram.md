# ТЗ: UI/UX Wordle для Telegram (текстовый интерфейс)

Документ описывает **единый UX/UI-стандарт** для реализации игры Wordle (5 букв) в Telegram-боте.  
Цель — максимально приблизить ощущение игры к Wordly/Wordle, **используя только текст, эмодзи и кнопки**, без canvas, webview и сложной графики.

Документ предназначен для прямой передачи в разработку.

---

## 1. Цели UX

Основные цели интерфейса:

- снизить когнитивную нагрузку пользователя;
- сделать состояние игры понятным с первого взгляда;
- визуально приблизить Telegram-версию к классическому Wordle;
- избежать «шума» и служебных сообщений;
- сделать игру ощущаемой как мини‑приложение, а не чат с ботом.

UX-ориентиры:
- пользователь **видит**, а не читает;
- буква и её статус — единое целое;
- одна игра = один визуальный блок.

---

## 2. Общий принцип интерфейса

### 2.1 Одна игра — одно сообщение
- Вся игра отображается **в одном сообщении**.
- После каждого ввода бот **редактирует сообщение**, а не отправляет новое.
- Чат не захламляется, пользователь всегда видит актуальное состояние.

### 2.2 Фиксированная структура
- Всегда отображается **полная сетка 6×5**, даже если попыток меньше.
- Пользователь заранее видит лимит попыток и прогресс.

---

## 3. Игровая сетка

### 3.1 Размер и структура
- 6 строк (попытки)
- 5 колонок (буквы)
- строки идут сверху вниз в порядке ввода

### 3.2 Состояния клеток

| Состояние | Эмодзи | Значение |
|----------|-------|----------|
| Пусто    | ⬜     | Попытка ещё не использована |
| Нет буквы | ⬛    | Буквы нет в слове |
| Есть буква | 🟨   | Буква есть, но в другой позиции |
| Верно    | 🟩    | Буква на правильном месте |

---

## 4. Формат отображения буквы

Каждая клетка отображается как:

```
<ЦВЕТ><БУКВА>
```

Пример строки:
```
⬛П🟩Р🟩У⬛Н🟩А
```

Правила:
- буквы всегда **заглавные**;
- пробелы не используются;
- цветной квадрат всегда стоит **перед буквой**;
- строка читается слева направо без декодирования.

---

## 5. Текущая попытка (якорь внимания)

Неиспользованная текущая строка должна быть визуально выделена.

Рекомендуемый формат:
```
➡️ ⬜⬜⬜⬜⬜
```

Требования:
- якорь отображается **только у одной строки**;
- якорь перемещается после каждой попытки;
- если игра завершена — якорь отсутствует.

---

## 6. Полный игровой экран (пример)

```
🎯 Слово дня #1444

⬛П🟩Р🟩У⬛Н🟩А
⬛С⬛Т🟩У⬛Ж🟩А
⬛К🟩Р🟩У⬛П🟩А
➡️ ⬜⬜⬜⬜⬜
⬜⬜⬜⬜⬜
⬜⬜⬜⬜⬜

Попытка: 3 / 6
```

---

## 7. Ввод и ошибки

### 7.1 Валидация ввода
Ошибки валидации:
- не 5 букв;
- не кириллица.

Формат ответа:
- кратко;
- без повторений;
- без разрушения основного сообщения.

Рекомендуется:
- отправлять отдельное краткое сообщение;
- не встраивать ошибки в игровую сетку.

### 7.2 Словарь
В мягком режиме:
- сообщения «слова нет в словаре» **не выводятся**;
- попытка всегда засчитывается.

Информация о словаре выносится в `/help`.

---

## 8. Финальные экраны

### 8.1 Победа

```
🎉 Победа!

🟩Ф🟩Р🟩А🟩З🟩А

Попыток использовано: 5 / 6
```

Требования:
- история попыток сохранена;
- сообщение финальное;
- появляется кнопка «▶️ Играть дальше».

---

### 8.2 Поражение

```
💥 Поражение

⬛П🟩Р🟩У⬛Н🟩А

⬛Л🟨У⬛Н🟩А

⬛П⬛Р⬛У⬛Н⬛А

Слово дня: ФРАЗА
```

---

## 9. Кнопки управления

Кнопки располагаются **под игровым сообщением**.

Обязательные кнопки:
- 🎮 Новая игра
- ℹ️ Инструкция
- 🏳️ Сдаться

Правила:
- кнопки доступны всегда;
- кнопки не создают новые игровые сообщения, кроме «Новая игра»;
- «Новая игра» создаёт новый игровой блок.

---

## 10. Принципы хорошего UX (обязательные)

- не дублировать служебные сообщения;
- не объяснять очевидное во время игры;
- не менять формат без причины;
- избегать «шумных» текстов;
- уважать паузы и финальность.

---

## 11. Критерии приёмки

Реализация считается корректной, если:
- пользователь с первого взгляда понимает состояние игры;
- история читается без сопоставления букв и цветов;
- вся игра живёт в одном сообщении;
- UI стабилен и предсказуем;
- UX субъективно близок к Wordly.
