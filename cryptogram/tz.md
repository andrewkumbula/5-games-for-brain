ТЗ — Криптограмма для Telegram-бота

1. Цель

Добавить в Telegram-бота ежедневную игру «Криптограмма».

Пользователь видит зашифрованную фразу, где каждая буква заменена кодом (числом). Нужно восстановить исходную фразу.

Игра доступна 1 раз в сутки.

⸻

2. Механика
	•	Каждая буква заменяется числом (кодом).
	•	Одинаковые буквы → одинаковые числа; ввод буквы для кода **K** заполняет **все** клетки с числом **K** (глобальная подстановка по коду).
	•	Часть букв открыта заранее (подсказки).
	•	Пользователь вводит буквы в остальные клетки.
	•	Ограничение ошибок: **3** (см. п. 9).

⸻

3. Ежедневная логика
	•	**1 игра на календарную дату** в часовом поясе **Europe/Moscow** (полночь по Москве — смена задания).
	•	Все пользователи получают **одну и ту же** фразу и **один и тот же** набор кодов (числа для букв генерируются **детерминированно от даты + id фразы**, см. п. 5).
	•	Прогресс сохраняется между сеансами.
	•	Завершённую игру (**won** / **lost**) нельзя начать заново в тот же день; при открытии показывается итог (и при поражении — разгадка).

⸻

4. Требования к фразам

Источник контента и расширенные правила — файл **`cryptogram/fraze.md`** (категории: поговорки, кино, книги, разговорные, доп.). Список там же считается **боевой базой** для ежедневной выдачи.

Обязательные правила текста:
	•	**минимум 3 слова** (фразы из 2 слов в прод не берём — слишком просто и мало повторов букв);
	•	**оптимально 3–8 слов**;
	•	**до 60 символов включительно** — считается **вся видимая строка фразы**: буквы, пробелы, **без учёта завершающих пробелов**;
	•	алфавит в фразе: **только русские буквы а–я (без «ё»)** и **пробелы** — буква **«ё»** в исходниках заменяется на **«е»**;
	•	**без** сложных/редких слов, **без** мата и политики;
	•	**без имён собственных** (по возможности);
	•	понятность широкой аудитории;
	•	**желательны повторяющиеся буквы** (усложняет и украшает логику).

Запрещено в прод-фразах: латиница, цифры, знаки препинания (запятые, тире, кавычки) — при подготовке `cryptograms.json` строки **нормализуются** к «буквы + пробелы». Слова с дефисом — заменить на пробел или переписать без дефиса (редакторский выбор в данных).

Примеры допустимого вида (после нормализации):
	•	Без труда не вытащишь и рыбку из пруда
	•	Делу время потехе час

⸻

5. Шифрование
	•	Каждой **уникальной букве** в фразе (после приведения к нижнему регистру) назначается **целое число-код**; одинаковые буквы — один код.
	•	Генерация **псевдослучайная, но воспроизводимая**: `seed = hash(date_YYYY-MM-DD + phrase_id)` (или аналог), затем случайное биективное сопоставление множества букв, встречающихся во фразе, с **подмножеством натуральных чисел** (например 1…N без дыр внутри диапазона для N букв — удобно для UI).
	•	**Пробелы не кодируются** — отображаются как есть.
	•	**Регистр не используется**: всё хранится и сравнивается как строчные буквы.
	•	В `cryptograms.json` на дату хранится минимум: `phrase_id`, нормализованный текст (или ссылка на элемент списка), опционально кэш кодов для отладки; **истина** — текст + seed-правила, чтобы клиент/сервер могли пересчитать коды.

⸻

6. Открытые буквы (подсказки)
	•	**2–5 букв** открыто с начала (только **разные позиции по буквенному ряду**, см. ниже).
	•	Распределение **по фразе** (не все подряд в начале): детерминированно от `seed`, чтобы у всех пользователей одинаково.
	•	Индексация **только букв**: строка `letters[]` — фраза без пробелов, длина `L`. Подсказки — множество индексов `0 … L-1`. Позиции с пробелами в этой модели **не существуют**.

⸻

7. Модель данных (состояние пользователя)

Идентификатор пользователя — как в боте (Telegram `user_id`).

```json
{
  "user_id": 123456,
  "date": "2026-03-27",
  "phrase_id": "proverbs_001",
  "guesses_by_code": { "3": "а", "7": "м" },
  "hint_letter_indexes": [0, 12, 22],
  "mistakes": 0,
  "status": "in_progress"
}
```

Поля:
	•	**date** — дата игры в **Europe/Moscow**.
	•	**phrase_id** — ключ фразы из каталога (связь с `cryptograms.json` / `fraze.md`).
	•	**guesses_by_code**: объект **код (строка или число в JSON) → буква**, какую букву пользователь сопоставил числу. Это единственный источник ввода; при вводе в одну клетку обновляются **все** клетки с тем же кодом.
	•	**hint_letter_indexes** — копия из задания дня (какие индексы в `letters[]` открыты сразу); в сохранении нужна для восстановления UI без повторного расчёта.
	•	**mistakes** — целое 0…3.
	•	**status**: `in_progress` | `won` | `lost`.

Старое поле `revealed_indexes` из черновика **заменяем** на **`hint_letter_indexes`** + явное описание в п. 6.

⸻

8. Канал UI: бот и мини-приложение

**Основная поставка:** та же **Telegram Mini App**, что «5 букв» и «Ассоциации» — игра **третьим пунктом в общем хабе** (`webapp`, карточка вместо «Игра 3»), одна сессия и те же приёмы (тулбар «← Меню», при необходимости `?date=` для тестов). Сетка «буква над числом» реализуется в **веб-интерфейсе** (вариант **A** ниже).

**Опционально / отдельно:** чистый **Telegram Bot API** (текст + кнопки) — если понадобится игра без WebView; тогда действуют ограничения ниже.

Ограничение бота: в одном сообщении **нет настоящей HTML-таблицы «буква над числом»** без костылей. Варианты (для не-web ветки или гибрида):
	•	**A)** Web App / Mini App — сетка как в макете, лучший UX (**основной путь для этого репозитория**);
	•	**B)** только бот: моноширинный блок (`<pre>`), строки вида букв/кодов, фокус по шагам через InlineKeyboard;
	•	**C)** гибрид: короткое описание в боте + кнопка «Открыть поле».

Макет экрана (сообщение в боте или аналог в Mini App):
	•	заголовок: «Криптограмма дня», счётчик «Ошибки: 0/3»;
	•	поле: клетки с буквами/пусто, под ними числа; подсказанные буквы нередактируемы;
	•	кнопки: **←** **→** (фокус по клетке), **Стереть**, **Проверить**, **Назад**.

Если реализация только Mini App — handlers в боте остаются точкой входа (`/cryptogram`, Web App URL), логика та же.

⸻

9. Логика ввода, проверки, победы и поражения

**Ввод**
	•	Фокус на одной **логической клетке** (позиция буквы в потоке букв + пробелы в отображении отдельно).
	•	Клавиатура / ввод только **одной русской буквы** за нажатие (а–я, **без ё** — при вводе ё нормализовать в е).
	•	Клетки подсказок и пробелы **не фокусируются** для ввода; **нельзя стереть/изменить** подсказанные буквы.
	•	**Стереть** — убирает букву у **текущего кода** (во всех клетках с этим числом), если эта клетка не подсказка.

**Навигация ← →**
	•	Переход между **редактируемыми** буквенными позициями в порядке слева направо по фразе; пробелы пропускаются; с края — по кругу **только среди редактируемых** клеток.

**Проверить** (основной момент подсчёта ошибок)
	•	Сравниваются **все коды**, для которых пользователь задал букву в `guesses_by_code`, с истиной по фразе.
	•	Для каждого кода **K**: если назначенная буква ≠ истинной букве для **K**, это **одна ошибка** (неверный код считается один раз, независимо от числа клеток с этим кодом).
	•	Коды, для которых буква ещё не назначена, **не дают ошибок** на этом шаге, но и **не засчитывают победу**.
	•	После подсчёта: если **mistakes ≥ 3** → **status = lost**, показать сообщение поражения и **полную расшифровку**; дальнейший ввод заблокирован.
	•	Если ошибок < 3 и все коды заполнены верно (см. победа) → **won**; иначе обновить счётчик ошибок и оставить `in_progress`.

Альтернатива «ошибка на каждую неверную клетку» **не используется** — иначе одна проверка может дать >3 ошибок.

**Победа**
	•	Условие: для **каждого** кода, встречающегося во фразе, в `guesses_by_code` стоит **верная** буква (подсказки уже совпадают с истиной). Эквивалентно: все буквы `letters[]` восстановлены верно.
	•	Можно вызвать проверку победы **после успешного «Проверить»** без ошибок и с полным заполнением, либо **сразу при заполнении последнего кода** — оба варианта допустимы, в коде зафиксировать один.

**Поражение**
	•	Три ошибки по правилу кодов выше.
	•	После **lost** показать полную фразу; повторной попытки в этот день нет.

⸻

10. Ограничения и краевые случаи
	•	Не более **одной буквы** за одно действие ввода в активную клетку.
	•	Нельзя менять буквы в **подсказках** (индексы из `hint_letter_indexes`).
	•	Нельзя повторно проходить завершённую игру за тот же **date**.
	•	Конфликт «одна буква — два разных кода» невозможен при модели **guesses_by_code**: пользователь задаёт букву **коду**, а не отдельной клетке.

⸻

11. Архитектура (черновик модулей)

**Mini App (этот репозиторий, общий цикл с «5 букв» / «Ассоциации»):**
	•	`webapp/cryptogram.js` — ядро логики, UI, `localStorage` по дате (паттерн как в `associations.js`);
	•	`index.html` — секция `#viewCryptogram`, тулбар и кнопка «К меню игр»;
	•	`hub.js` — `showCryptogram`, `hideGameViews`, карточка `data-game="cryptogram"`, хеш `#cryptogram`;
	•	`styles.css` — стили сетки криптограммы;
	•	данные: `cryptogram/cryptograms.json` (или рядом с игрой), загрузка через `fetch`.

**Серверный Telegram-бот (отдельный стек, по желанию):**  
handlers/cryptogram.py, services/cryptogram_service.py, repositories/cryptogram_repository.py, models/cryptogram.py.

⸻

12. Хранение
	•	**cryptograms.json** — каталог фраз и `phrase_id`; связь **дата → phrase_id** по правилам расписания (ротация по списку из `fraze.md`).
	•	**Mini App:** прогресс в **localStorage** (ключ вида `cryptogram:<дата>` или единый префикс с версией, как в других играх).
	•	**SQLite** — при реализации **серверного** бота и общей статистики; для одного только Mini App не обязателен.

⸻

13. Что реализовать
	•	модели данных в JS + сохранение в localStorage (и при необходимости миграции SQLite для бота);
	•	загрузка задания по дате (Москва) и `phrase_id`;
	•	детерминированная генерация кодов из seed;
	•	выбор подсказок (2–5 индексов в `letters[]`);
	•	ввод / навигация / стереть / проверить;
	•	подсчёт ошибок по кодам, победа / поражение;
	•	сохранение и восстановление прогресса;
	•	нормализация фраз из контент-пула под правила п. 4.

⸻

14. Acceptance Criteria
	•	игра открывается на текущую дату (Москва);
	•	виден шифр и подсказки;
	•	ввод и глобальная подстановка по коду работают;
	•	ошибки считаются по правилам п. 9;
	•	есть победа и поражение с блокировкой повтора;
	•	прогресс сохраняется между перезапусками;
	•	фразы соответствуют `fraze.md` и п. 4.

⸻

15. MVP
	•	1 игра в день (Москва);
	•	JSON каталог + localStorage в Mini App;
	•	хаб + экран игры в Web App; бот без сетки — опционально.

⸻

16. Краткая формулировка задачи

Реализовать игру «Криптограмма»: фраза дня из контент-пула, шифрование букв числами с воспроизводимым seed, частично открытые буквы, ввод с подстановкой по коду, лимит 3 ошибок (по неверным **кодам** при проверке), победа / поражение с показом ответа при проигрыше, сохранение прогресса, единые правила фраз (`fraze.md`).

⸻

17. Согласование с `cryptogram/fraze.md`

Требования к фразам в настоящем ТЗ **расширяют и заменяют** старый пункт «2–5 слов» из черновика: актуальны **минимум 3 слова**, **до 60 символов** с пробелами, **без «ё»**, акцент на **повторах букв** и отсутствии лишних символов. Наборы строк в `fraze.md` использовать как основной источник; перед включением в JSON выполнять **нормализацию** (ё→е, удаление пунктуации, проверка длины и слов).
